# Proposed Reorganization for src/components

Based on an analysis of the current 50+ files in `src/components`, the following structure is recommended to improve scalability, maintainability, and clarity. This structure separates atomic UI components from landing page sections and feature-specific logic.

## 1. **Directory Structure**

```text
src/components/
├── common/ (or ui/)        # Atomic, reusable components (mostly from shadcn/ui)
│   ├── buttons/            # If you have many button variants
│   ├── data-display/       # Table, Badge, Avatar, TruncatedText
│   ├── feedback/           # Alert, Modal, Toast, LoadingSpinner
│   ├── forms/              # Input, Select, Textarea, Label
│   └── navigation/         # NavigationMenu, Tooltip
├── layout/                 # High-level layout components used across pages
│   ├── Header.tsx          # Main site header
│   ├── Footer.tsx          # Main site footer
│   └── Navigation/         # Mobile/Desktop menus, MenuNavigation.tsx
├── sections/               # High-level sections (mostly for the Landing Page)
│   ├── Hero/               # Consolidate Hero.tsx, hero/, and Hero/ folders
│   ├── Benefits.tsx        # "Vantagens" section
│   ├── CoursesSection.tsx  # Landing page courses list (Rename from Courses.tsx)
│   ├── Instructors.tsx     # "Instrutores" section
│   ├── StatsSection.tsx    # Statistics and numbers
│   ├── TestimonialCarousel.tsx # User feedback/reviews
│   └── ProjectInfo.tsx     # About the project sections
└── features/               # Domain-specific components and complex logic
    ├── courses/            # Course-related logic (cards, details)
    │   ├── CourseCard.tsx
    │   ├── LessonCard.tsx
    │   └── loading/        # Course-specific loading states
    ├── dashboard/          # All components specific to user dashboards
    ├── subscription/       # Plans, pricing, and subscription status
    ├── video/              # VideoPlayer and related controls (Move from ui/)
    └── auth/               # UserForm.tsx, LoginForm, etc. (From current forms/)
```

## 2. **Key Consolidation & Cleanup**

### A. Resolve `Hero` / `hero` Redundancy
Currently, there are `hero/`, `Hero/`, and `Hero.tsx` at the root.
- **Action**: Merge all into `src/components/sections/Hero/`.
- **Reason**: Case-sensitive folders can cause issues in Git/CI and are confusing for developers. Use a consistent casing (e.g., PascalCase for component folders).

### B. Move VideoPlayer out of `ui/`
`VideoPlayer.tsx` is currently in `src/components/ui/`.
- **Action**: Move it to `src/components/features/video/`.
- **Reason**: `ui/` should ideally contain atomic components. `VideoPlayer` is a complex feature component with its own logic, sub-components (QualitySelector, SubtitleSelector), and styles.

### C. Standardize Loading Components
You have `LoadingScreen.tsx`, `LoadingSpinner.tsx`, `CourseDetailLoading.tsx`, and `CoursePageLoading.tsx`.
- **Action**: Move generic loaders to `src/components/common/loading/` and domain-specific ones to `src/components/features/courses/loading/`.

### D. Separate Forms from Feature Logic
Existing `src/components/forms/` contains both `DataTable.tsx` and `UserForm.tsx`.
- **Action**:
    - Move `DataTable.tsx` to `src/components/common/data-display/`.
    - Move `UserForm.tsx` and `LessonForm.tsx` to their respective features (e.g., `features/auth/` or `features/courses/`).

## 3. **Benefits of this Approach**

1.  **Easier Navigation**: Developers can quickly find where a component lives by its "type" (Is it atomic? Is it a section? Is it domain-specific?).
2.  **Scalability**: New features can be added in `features/` without cluttering the root folder.
3.  **Encapsulation**: Domain-specific logic stays near the components that use it (e.g., all things "Course" are grouped).
4.  **Consistency**: Resolves naming and casing conflicts that currently exist in the folder structure.
